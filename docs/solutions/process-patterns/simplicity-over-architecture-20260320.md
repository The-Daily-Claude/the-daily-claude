## Simplicity Over Architecture for Content Pipelines

### The Lesson
The initial V1 plan had:
- 6 phases (13-18 sessions)
- Hexagonal architecture with port traits and use case layers
- Multi-model review panel with 5-axis scoring
- Webhook server with HMAC verification
- 3 platform-specific image sizes
- 3 platform-specific caption styles
- UUIDv7 + freshness tracking for 122 entries

The simplicity reviewer's verdict: "This plan architectures a meme account like a production microservice."

### What Was Cut
| Original | Simplified To |
|---|---|
| 6 phases, 13-18 sessions | 3 phases, 6-9 sessions |
| Hexagonal architecture | Flat modules (6 files) |
| Multi-model review panel | One model, pass/fail |
| Webhook server | Poll Linear on cron |
| 3 image sizes | 1080x1080 everywhere |
| 3 caption styles | One caption ≤280 chars |
| UUIDv7 + freshness tracking | Boolean `used` field → then markdown files with YAML frontmatter |
| Cargo workspace with port traits | Single crate, flat modules |

### What Shipped First
Day 1: manually post a meme generated by one curl command to memegen.link.
Everything else is automation of what you're already doing manually.

### The Rule
The simplest version that ships content TODAY beats the most elegant version that ships in three weeks. Add complexity only when real usage proves it's needed.

### Exception
Anonymization was NOT simplified — it's safety-critical. Personal info + credentials on 4 social platforms simultaneously = real damage. The review model's safety veto was kept.
